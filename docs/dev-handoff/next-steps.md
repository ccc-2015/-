# 下一步开发路线

## 优先级 1：知识库检索质量

目标：把当前本地 `local_hash_v1` 混合检索升级为更接近生产的可解释检索链路。

建议步骤：

1. 按 `docs/dev-handoff/embedding-plan.md` 接入真实 embedding provider。
2. 将 `knowledge_embeddings.vector_json` 迁移为 PostgreSQL + pgvector，停止用 JSON 扫描向量。
3. 将 `local_hash_v1` 作为 fallback，默认检索切到真实 embedding provider，并保留关键词混合召回。
4. 继续增强语义相似去重，减少同源政策、院校章程不同年份之间的误判。

验收点：

- 管理端能查看每份文档的清洗质量报告和问题列表。
- 低质量文档发布前有明确拦截或复核提示。
- 完全重复的正文或来源 URL 不能发布。
- Agent 引用中包含 `document_id`、`chunk_id`、`version`、`score_detail`，并在用户端可见。
- 不发布的文档不会被 Agent 检索。

## 优先级 2：志愿方案增强

目标：在已保存方案的基础上，让用户可以更精细地调整、复用和导出志愿方案。

建议步骤：

1. 将当前 JSON 导出升级为表格或网页结构化导出。
2. 后端补充统一数据版本引用，导出、二次校验和报告生成都记录同一批数据版本。
3. 如需要更强交互，再把当前上移/下移排序升级为拖拽排序。
4. 补充多方案对比视图，便于考生比较不同风险偏好的版本。

验收点：

- 用户能保存多个同批次方案版本。
- 导出内容使用当前方案的顺序和快照。
- 二次校验、报告生成和导出都使用保存后的顺序和 `group_id`。

## 优先级 3：报告生成

目标：把画像、推荐、方案、政策引用汇总成可分享报告。

建议步骤：

1. 将网页报告导出为 PDF 或表格文件。
2. 报告中的政策依据接入 published 知识库引用，而不是固定规则说明。
3. 补充报告分享链接或只读查看页。
4. 报告生成时记录统一数据版本，覆盖推荐数据、知识库引用和方案版本。

验收点：

- `/user/reports` 不再是静态数据。
- 报告能从已保存方案生成。
- 报告记录生成时间和数据版本。

## 优先级 4：管理端运营闭环

目标：让管理员看到真实运营状态。

建议步骤：

1. `/admin` 指标改为真实 API。
2. `/admin/chat-logs` 接 agent_conversations、agent_messages、tool_calls、citations。
3. `/admin/agent-ops` 展示节点运行次数、失败次数、工具命中。
4. `/admin/audit-logs` 增加审计表和接口。

验收点：

- 管理员能查看真实 Agent 会话。
- 能看到每次回答调用了哪些工具和引用了哪些文档。

## 优先级 5：自动化测试

目标：减少每次改动后的回归风险。

建议步骤：

1. 后端加 pytest。
2. 覆盖：
   - 登录
   - 画像保存
   - 数据导入
   - 推荐生成
   - 政策校验
   - 知识库上传和切片
   - Agent 推荐/政策意图
3. 前端先保持 `npm run type-check`，后续再加页面级测试。

验收点：

- 新增后端测试命令。
- CI 或本地一条命令可跑核心接口测试。
